Accessibility Challenges for Web Interaction

  • Web 1.0
    • Web pages are static documents
    • Interaction limited to simple form controls
  • Web 2.0
    • Web pages are dynamic
    • Interaction with custom complex widgets

WAI-ARIA

  • AT need to ask the browser what an element is
  • Browsers need to identify that element and its associated properties in a meaningful way
  • ARIA provides this
    • Map leaf level widgets to their equivalents in traditional desktop applications
    • When there is an update, give the AT some hints about what should be spoken
    • DOM + JS is the assembly language of the web - ARIA adds a few special operators to this

WAI-ARIA

  • Roles (with related properties)
    • "slider" role for DIVs that have JavaScript + CSS which act as sliders
    • AT will see these DIVs as slider widgets in a desktop app
  • Live Regions
    • Regions with dynamic content
    • Hint for how updates to these regions should be processed

WAI-ARIA

  • Experiencing Web Apps with ARIA
    • Web app uses ARIA
    • ARIA aware browser + AT combination
    • Turn off browse mode so that you can interact with the web application as an application and not as a document

Access Enabling UI Libraries

  • Google Web Toolkit
    • Enables web developers to program in a higher level language (Java)
    • Compiles it to the assembly language of the Web (HTML DOM and JS)
    • Accessible building blocks for GWT 1.5

Gap Analysis

  • Web applications are more than the sum of their leaf level widgets
  • The AT user has to drive two applications simultaneously; the web app to do the task and the screen reader to discover what the web app has done
  • Visually impaired users should use the application the same way as a sighted user and get feedback through their AT
  • Script enhanced accessibility for a smoother user experience

Gap Analysis

  • Creating a good user experience for traditional desktop apps
  • Start with the accessibility API so that all the widgets are accessible
  • Use an application specific script to handle higher level logic
    • Word processors
    • Calendar applications

Gap Analysis

  • Script enhanced accessibility can be made available by:
    • Web application providers injecting the needed augmentation based on user preferences. Such injection has to be independent of any given adaptive technology to avoid a combinatorial explosion.
    • AT vendors injecting such augmentation for their AT.
    • Web user agents relying on metadata to identify relevant portions of a Web user interface, and exposing such metadata via reflection to the adaptive technology.
      • Higher level information
      • Outlook mail folders, not individual checkboxes
    • The Web community at large experimenting with different approaches that work at Web scale. Such large-scale experimentation can be used to discover and codify design patterns that work as a means of arriving at the next generation of accessibility techniques.

Google Reader

  • Feed reader
  • Web feeds - updated content from websites
  • Advantages for the user:
    • Only subscribe to feeds that are of interest
    • Just the content without the clutter
    • One place to look for updates instead of going to the individual sites

Google Reader

Google Reader

  • More akin to a desktop application than a web page
  • Users need to be able to do more than just identify the individual widgets
  • Reader is already a keyboard driven app
  • Users should automatically hear the current feed/article when they navigate to it

Google Reader

  • Initially released as a script for experimentation
  • Enhanced version is now integrated and available as part of Reader
  • General pattern: Experiment and discover what is useful, take lessons learned and integrate back into product

Google Search

  • AJAX improving usability
  • Press a key to go to the next result
  • When on the last result, that same key loads the next page and goes to the first result

Gmail

Gmail

Gmail

  • Also a keyboard driven app
  • Speak the sender, subject, snippet, and timestamp when the user goes to the next/previous message in the inbox
  • Read the message when the user moves to the next/previous message in the conversation
  • Announce alerts ("Your message has been sent.") when they appear
  • Speak instant messages from the chat system inside Gmail
